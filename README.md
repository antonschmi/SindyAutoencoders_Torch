# SindyAutoencoders (PyTorch)

A modern, GPU-friendly PyTorch reimplementation of `../SindyAutoencoders`
(Champion, Lusch, Kutz & Brunton, ["Data-driven discovery of coordinates and
governing equations"](https://arxiv.org/abs/1904.02107)). Same architecture,
same SINDy library, same loss terms, same sequential-thresholding procedure -
just a different framework and a training loop that actually keeps a GPU busy.

## Why this exists

The original code is TensorFlow 1 (`tf.placeholder` / `tf.Session` / manual
per-activation derivative formulas). Porting it to run under TF2's
`tf.compat.v1` shim (see `../SindyAutoencoders`) got it running again, but
profiling showed GPU acceleration barely helped: the training loop calls
`sess.run(..., feed_dict=...)` once per mini-batch (~250 calls/epoch for the
Lorenz example), and that per-call Python/session dispatch overhead - not the
actual matrix math - dominates wall-clock time. Moving the same loop structure
to a GPU just moves a cheap computation closer to an idle accelerator.

This version fixes the actual bottleneck:

- **Data lives on the device for the whole training run.** The full
  train/validation tensors are moved to the GPU once, up front. Batches are
  just slices of an already-resident tensor, so there's no per-step
  host<->device transfer at all.
- **Derivatives via forward-mode autodiff, not hand-derived formulas.** The
  original computes `dz/dt` and `d²z/dt²` by manually propagating the chain
  rule through each activation function (separate formulas for `elu`, `relu`,
  `sigmoid` x order 1, order 2). This version uses `torch.func.jvp` (nested
  for second order) instead - exact for any activation, and it removes six
  hand-transcribed formula variants as a source of bugs. Verified against
  finite differences and cross-checked numerically against the original
  TF1 implementation with identical weights (see below).

## Structure

```
src/
  sindy_utils.py   - library_size, sindy_library, sindy_simulate, etc. (pure numpy, copied unchanged)
  model.py         - SindyAutoencoder (nn.Module): encoder/decoder, SINDy library, loss
  training.py      - GPU-resident training loop, sequential thresholding, checkpointing
examples/
  lorenz/          - data.py, train_lorenz.py, analyze_lorenz.py
  pendulum/        - data.py, train_pendulum.py, analyze_pendulum.py  (model_order=2)
  rd/              - data.py, train_rd.py, analyze_rd.py  (see caveat below)
```

## Running

From inside an example's own directory (matches the original notebooks'
convention - relative imports assume this):

```bash
cd examples/lorenz
python train_lorenz.py      # trains, saves <name>.pt + <name>_params.pkl + a results dataframe
python analyze_lorenz.py model1   # loads a checkpoint and reports reconstruction/SINDy error
```

Same pattern for `pendulum/`. `train_*.py` auto-detects a GPU
(`torch.cuda.is_available()`); on Colab, just `Runtime -> Change runtime type
-> GPU` first.

## Deliberate differences from the original

- **`params['shuffle']`**: defaults to `False` to match the original's fixed,
  non-shuffled sequential batching exactly (useful if you're trying to
  reproduce the paper's numbers as closely as possible). Set to `True` for
  more standard per-epoch shuffling.
- Checkpoints are a single `<name>.pt` (`torch.save`) instead of the TF
  checkpoint triplet (`.data`/`.index`/`.meta`); `<name>_params.pkl` is kept
  for parity with the original layout.

## Known gap: reaction-diffusion data

`examples/rd/data.py` expects `reaction_diffusion.mat` in that directory,
generated by the MATLAB solver in `../SindyAutoencoders/rd_solver/`. That file
isn't part of either repo, so `train_rd.py`/`analyze_rd.py` are ported
faithfully but untested end-to-end - I validated the model/training code path
itself using synthetic data of the right shape, but real verification needs
that `.mat` file.
